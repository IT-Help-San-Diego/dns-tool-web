Our Approach

DNS Tool operates with the same rigor expected of any instrument that produces findings people act on. Every design decision traces to a standard. Every claim traces to code. Every result can be independently verified.

We Implement the Standards as Written

Our analysis engine doesn't invent interpretations. SPF evaluation follows RFC 7208. DMARC alignment follows RFC 7489. DANE/TLSA follows RFC 6698. DNSSEC validation follows RFC 4033–4035. Certificate Transparency follows RFC 6962. MTA-STS follows RFC 8461. TLS-RPT follows RFC 8460.

When we tell you something about your domain's security posture, we can point to the exact RFC paragraph that defines the behavior we're evaluating.

RFC 7208 RFC 7489 RFC 6698 RFC 4033–35 RFC 6962 RFC 8461 RFC 8460 RFC 8659 RFC 7001

We Show Our Work

Every analysis includes "Verify It Yourself" terminal commands — dig, openssl, curl — so any analyst can independently reproduce our findings. We publish our confidence taxonomy, our source hierarchy, and our methodology. Our intelligence sources inventory shows exactly where every data point originated.

If we cannot show you how to verify a claim, we should not be making it.

"Go out and gather as many different redundant sources of intelligence as you can, and then classify and analyze."

Why This Matters

We've received feedback that this level of rigor is "too much" for a DNS checker. That using Lighthouse, Mozilla Observatory, and scientific methods for color selection is excessive. We understand the sentiment — most tools in this space don't do these things.

We think that's the point.

These aren't vanity metrics. They're the validation tools built by the organizations that define web quality standards. Lighthouse was built by Google's Chrome team. Mozilla Observatory was built by the organization that maintains Firefox. SonarCloud is an industry-standard static analysis platform. Using them isn't excessive — it's the baseline that any tool producing security findings should meet.

The biggest brands in the world routinely fail these tests. Their Lighthouse scores are in the 40s. Their security headers are incomplete. Their accessibility is an afterthought. The fact that a DNS security tool passes them all shouldn't be surprising — it should be expected. We just happen to be one of the few that actually does it.

If being thorough is what makes us different, we'll take that as confirmation we're doing it right.

Transparency by Default

We don't hide behind black-box analysis. Our reports carry TLP classification under FIRST TLP v2.0. Our confidence engine explicitly distinguishes between what we've directly observed, what we've inferred from patterns, and what came from third-party enrichment. Every attribution is classified:

  • Observed — Directly witnessed in authoritative data
  • Inferred — Derived from patterns in primary data
  • Third-party — Sourced from external enrichment

The consumer always knows the basis for each finding. If there's a better way to produce trustworthy security intelligence, we want to learn it.

A Note from the Builder

I don't have a computer science degree. I'm self-taught. Everything in this platform — every RFC implementation, every scoring engine, every quality gate — was learned from reading the standards, studying the science, and applying it with the discipline of someone who couldn't afford to get it wrong.

This project exists because I believe security intelligence should be transparent, reproducible, and grounded in evidence. Not because it's trendy. Because it's the right way to build tools that people rely on to protect their organizations.

We show our work. We show our thought processes. We hold ourselves to the standards that exist specifically to measure quality. If that seems like too much — consider what the alternative looks like.
